비동기로 하는 프로그래밍 공부_01 : VR Async Box Controller (Meta SDK + UniTask)

기술 스택
- Engine: Unity
- SDK: Meta Interaction SDK (`Oculus.Interaction`)
- Async Library: UniTask (`Cysharp.Threading.Tasks`)

에디터 세팅 방법

1. **오브젝트 구조 분리**
   - 빈 게임 오브젝트(부모)를 생성하여 `BoxController.cs`를 부착합니다. (Z축 파란색 화살표가 다가올 방향을 향해야 함)
   - 실제 모델링(자식)은 부모 아래에 넣고 로컬 포지션을 `(0,0,0)`으로 맞춥니다.
2. **버튼 연동**
   - 씬에 Meta SDK의 `Poke Button` 프리팹을 배치합니다.
   - `BoxController`의 Inspector 창에서 `PokeInteractable`과 `InteractableUnityEventWrapper` 필드에 버튼 오브젝트를 드래그 앤 드롭으로 연결합니다.
3. **내부 값 설정**
   - `Move Distance`: 앞으로 튀어나올 거리 (예: 0.5f = 50cm)
   - `Move Duration`: 이동에 걸리는 시간 (예: 1.0f = 1초)


비동기 프로그래밍을 활용한 이유

	1. 코드가 더 간략해 져서 보기 편하다
	2. 기능을 하나를 수정하거나 멈추려고 하면 우선적으로 작업이 끝나기까지 대기하거나 여기저기 에러가 터지기도 했는데
	비동기 프로그래밍과 CancellationToken을 활용해보니, 작업이 끝나기를 마냥 기다릴 필요 없이 원할 때 즉시 멈출 수 있고, 멈출 때 필요한 뒷정리도 한 곳에서 깔끔하게 처리할 수 있었다.
	(물론 동기(Sync)와 비동기(Async)가 다른 개념이긴 하지만, 동기(Sync)는 한 작업이 끝날 때까지 멈춰서 기다리는 것이면 비동기(Async)는 오래 걸리는 작업을 백그라운드에 두고 게임은 계속 부드럽게 돌	아가는 방식의 차이가 있긴 하다.)
	3. 제일 중요하지만 공부란 새로운 것을 도전하는 것이기 때문에 새로운 방식을 공부하기로 했다.

주요 기능
Meta SDK에 있는 **Poke**를 활용하여 VR 기기에서 버튼 하나로 오브젝트의 전진/복귀 상태를 제어했습니다. `CancellationToken`을 활용하여 이동 도중 복귀 명령이 들어와도 겹치거나 튀는 현상 없이 즉시 방향을 전환하게끔 구현했습니다.

### 1. CancellationToken이란? 
비동기 작업(`UniTask`, `Task`)은 한 번 실행되면 백그라운드에서 끝까지 완료하려는 성질이 있습니다. 만약 작업 도중 유저가 취소 버튼을 누르거나 화면을 나가버렸을 때, 이 작업을 강제로 멈추지 않으면 메모리 낭비나 치명적인 충돌(버그)이 발생합니다. `CancellationToken`은 이러한 비동기 작업에 **브레이크(원격 정지)** 역할을 합니다.

비동기 취소 시스템은 크게 명령자와 수행자 두 가지로 나뉘는데 비유를 들자면 다음과 같습니다.
* **CancellationTokenSource (CTS) = 고객(명령자)**
  * 배달을 시킨 후 마음이 바뀌면 취소할 수 있는 권한을 가진 주체입니다.
  * 주문 후 취소한다고 전화를 하는 행위가 코드 상으론 `cts.Cancel();` 입니다.
* **CancellationToken (Token) = 배달 기사 (수행자)**
  * CTS로 발급받아 비동기 함수에게 쥐여주는 연락망(수신기)입니다. 
  * 기사님이 배달을 하며(루프를 돌면서) 취소가 됐는지 안됐는지 지속적으로 확인하는 행위가 코드 상으론 `token.ThrowIfCancellationRequested();` 입니다.실제 코드를 작동 흐름을 보면

### 2. 실제 코드 작동 흐름

```csharp
// 1. [고객] 취소 스위치(CTS) 생성 및 연락망(Token) 발급
cts = new CancellationTokenSource();
var token = cts.Token;

// 2. [기사님] Token을 들고 비동기 작업(이동) 시작
try 
{
    while(이동 중...) 
    {
        // 3. [기사님] 매 프레임마다 취소 연락이 왔는지 확인
        // 연락이 왔다면 여기서 즉시 에러를 던지고 탈출!
        token.ThrowIfCancellationRequested(); 
        
        // 실제 이동 로직...
        await UniTask.Yield(PlayerLoopTiming.Update, token);
    }
}
catch (OperationCanceledException) 
{
    // 4. [기사님] 취소 연락을 받고 안전하게 오토바이를 세움 (뒷정리)
    Debug.Log("작업이 안전하게 취소되었습니다.");
}

// 5. [고객] 마음이 바뀌어 다른 버튼을 눌렀을 때 취소 명령 내리기
cts.Cancel();  // 작업 중지!
cts.Dispose(); // 메모리 정리 완료

'''

이런식으로 작동을 하게 되는데 요약하자면 하나의 중앙 스위치(CancellationTokenSource)를 만들고, 거기서 뻗어 나온 선(CancellationToken)을 비동기 함수에 연결하여 언제든 폭파(강제 종료)시킬 수 있게 만드는 시스템 이라고 보면 되겠습니다.

부드러운 이동과 이벤트 연결
UX 최적화: VR을 활용하여 작업된 내용을 확인해야 하기에 갑자기 확 이동하면 멀미가 생긴다는 것을 고려했습니다. 따라서 Mathf.SmoothStep을 적용하여 멀미를 줄이고 부드러운 감가속 이동을 구현했습니다.

이벤트와 비동기의 연결 (.Forget): 버튼 클릭과 같은 유니티 이벤트(AddListener)는 반환형이 없는 일반 함수만 연결할 수 있습니다. 이를 해결하기 위해 일반 함수 내부에서 비동기 함수를 호출하되, 결과 대기로 인한 병목을 막고 에러를 안전하게 로깅할 수 있도록 .Forget()을 명시적으로 붙여 백그라운드 작업으로 위임했습니다.

핵심 로직 및 새롭게 공부하며 배운 점

비동기 프로그래밍(UniTask)을 사용후 깨달음
이번 상호작용 로직을 구현하면서 기존의 동기(Sync) 방식이나 코루틴 대신 비동기(Async) 프로그래밍을 새롭게 도입해 보았습니다. 큰 장점을 체감했습니다.
일단 첫번째론 코드가 훨씬 간략해져서 가독성이 크게 향상되었습니다.
그리고 새로운 경험을 했습니다. 기존에 동기(Sync)로 작업하면 기능 하나를 수정하거나 멈추려고 하면 우선적으로 작업이 끝나기까지 대기해야 하거나 여기저기서 꼬여서 에러가 터지기도 했습니다. 하지만 비동기(Async)와 CancellationToken을 활용해보니, 작업이 끝나기를 마냥 기다릴 필요 없이 원할 때 즉시 멈출 수 있고, 멈출 때 필요한 뒷정리도 한 곳(catch 블록)에서 깔끔하게 처리할 수 있었습니다.
동기(Sync)는 작업이 끝날 때까지 멈춰서 기다리는 것이고, 비동기(Async)는 오래 걸리는 작업을 백그라운드에 두고 게임은 계속 부드럽게 돌아가게 하는 방식이라는 차이를 확실히 이해했습니다. 무엇보다 "공부란 새로운 것을 도전하는 것"이라는 생각으로 새로운 방식을 파고들었습니다. 아직 공부는 더 해야된다는 생각이 들어 계속 공부 할 예정입니다.
```
